Shape of the thing

A single Express server (server.js, roughly 1,200 lines) serving static HTML pages and a JSON API. No database, no build step, no framework, no bundler. Data lives in JSON files on disk. Each page is a self-contained HTML file with its own inline <style> and <script>.

That means you can edit any page and reload — nothing to compile. It also means duplication: the nav markup, the theme system and the CSS variable block are copied into every page. Changes to those must be applied everywhere, which is usually done with a small script rather than by hand.

Node 18+ is required (the start script uses --env-file-if-exists). Dependencies: express, express-session, bcryptjs, chess.js.

Storage

Six JSON files, each read and written whole. All resolve to process.env.DATA_DIR when set, falling back to the repo directory:

FileShape
users.json{ [lowercaseName]: user } — the username is the key
games.jsonChess, keyed by the non-hermione player
deathroll.jsonDeathroll, keyed the same way
mail.jsonThreads, keyed the same way
writing.jsonArray of categories, each holding passages
site.jsonThe editable dashboard copy

All six are gitignored. Production keeps its own copies on a Railway volume, so local testing can never affect live data — and a deploy never overwrites live data either.

The user record

Written keys: username, passwordHash, createdAt, pronouns, rank, points, dollars, icon, bio, flagged, notifications, plus per-day bookkeeping: lastCheckIn, wheelDay, snakeDay, snakeToday, dailiesNotifiedOn.

Everything is read defensively (user.dollars || 0) because records predate most fields. Adding a field needs no migration; removing one means tolerating stale keys. Rank does this already — rankFor() still maps the retired role values domme/sub onto Visitor/Citizen.

Read-modify-write. Every handler loads the whole file, mutates, and saves. Node's single thread plus synchronous fs makes each handler atomic, so concurrent requests can't interleave — but two handlers that both load users must not run await between load and save, or the later write will clobber the earlier.

Auth and permissions

Passwords are bcrypt-hashed (cost 10). Sessions come from express-session with an in-memory store — a server restart signs everyone out, which is fine here but is also why the daily check-in can't depend on login alone.

There is exactly one permission check in the whole codebase:

isAdmin(req) — true when the session username lowercases to "hermione".

Admin is an identity, not a flag on the record, so it can't be granted by editing data. Every privileged route calls it server-side. The client hides admin controls too, but that's cosmetic — the server is the boundary.

Permission patterns worth knowing:

  • canEditProfile() — the owner or Hermione
  • Rank changes — Hermione only, even on your own profile
  • Games keyed by player — the key is the authorisation; a player can only ever address their own row

Front-end conventions

Theming

Every page declares the same CSS variables twice: once in :root for light, once in body[data-theme="dark"] for dark. Components reference variables, never literals — a hardcoded colour is a bug waiting for the next palette change, and the greyscale conversion had to hunt down a dozen of them.

Theme selection is deliberately three-state: no stored preference means follow the system and keep following it; clicking the toggle stores a choice that wins from then on. applyTheme(theme, persist) only writes to localStorage when persist is true — page load must never persist, or the site stops tracking the system.

Shared scripts

notifications.js and ranks.js inject their own styles and markup into whatever page includes them, rather than each page carrying copies. notifications.js finds .top-nav .nav-links and appends the bell and envelope; it polls every 15 seconds and rides that one request for both notification content and the mail unread count.

The meow is synthesised with Web Audio — a sawtooth swept 620→880→500 Hz through a moving bandpass formant, with vibrato. No audio file to host, and it fails silently if the browser hasn't allowed audio yet.

Rendering

DOM is built with createElement and textContent, not innerHTML with interpolated values. Usernames, bios and messages are all user-controlled; textContent makes injection structurally impossible. The one place innerHTML appears is for fixed SVG icon markup.

The day key

Every daily thing hangs off todayKey(), which returns a YYYY-MM-DD string that rolls over at noon America/New_York, not midnight. It reads the Eastern wall clock via Intl.DateTimeFormat and subtracts 12 hours before taking the date, so the label only changes at midday. Using the zone rather than a fixed offset means it survives daylight saving.

Anything daily compares a stored key against this one. Testing a rollover is a matter of backdating the stored key rather than waiting.

Don't compute day keys any other way. A plain toISOString().slice(0,10) disagrees with this function for twelve hours out of every day — which is exactly how one test appeared to fail when the code was correct.

Currency and trust

Points are admin-granted only. Dollars are earned, and two of the three sources are browser-reported, which can't be trusted outright. The defences are deliberately economic rather than cryptographic:

  • Daily cap — Snake pays at most $20/day, so farming gains nothing over honest play.
  • Token bucket — 8 pickups burst, refilling one per 2s. A fixed minimum gap was tried first and rejected: food can legitimately spawn in front of the snake and be eaten on the next 120ms tick, so a hard floor would reject honest pickups.
  • Server-side outcomes — the wheel and deathroll pick their results on the server; the client only animates what it's told.

If dollars ever buy something worth cheating for, the real fix is server-authoritative gameplay — the client sends inputs, the server owns the state. Until then the cap is the honest boundary.

Game implementations

Two-player games (chess, deathroll) are keyed by the non-Hermione player and polled every 3–4 seconds. Deathroll additionally pushes a notification on each roll, using a fixed id per game so repeat rolls update one line rather than stacking twenty.

Traps this codebase has already sprung

Each of these cost real debugging time. They will recur.

  • hidden loses to display. An element styled display: flex ignores the hidden attribute. This leaked the admin nav dropdown to every user, and the deathroll opponent picker to players. Any component that might be hidden needs an explicit [hidden] { display: none; } rule.
  • Buttons don't inherit fonts. A <button> falls back to the browser UI font unless given font-family: inherit. That's why "Games" looked wrong next to its neighbours.
  • Hover gaps break dropdowns. A menu offset below its toggle leaves dead space the cursor must cross. A transparent ::after bridging the gap keeps :hover alive.
  • CSS specificity in grids. .dr-grid > .card (two classes) beats .player-card (one), which silently sized the deathroll side cards like the board.
  • Selects can't show a value they don't have. Setting a <select> to a value with no matching option leaves it blank — legacy free-text pronouns would have silently cleared on save, which is why values are canonicalised server-side.
  • Two callers, one-shot data. The dashboard once called /api/me twice; whichever landed first consumed the once-per-day check-in result and the notice vanished at random.

When editing many pages at once

Sweeping edits across the ten HTML files are usually scripted. Prefer Python with explicit string replacement over perl -pi -e: a malformed perl substitution prepended text to every line of dashboard.html and required restoring the file from git. Always verify with a grep count afterwards, and keep the working tree committed before a bulk edit.

Recipes

Adding a page

  1. Copy an existing view — the shell (variables, nav, theme script) is the point.
  2. Add app.get("/thing", requireLogin, …) pointing at it.
  3. Add the nav link to every page, scripted.
  4. Include /notifications.js so the bell and mail follow.

Adding a daily objective

Append an object to the array in GET /api/dailies with label, detail, reward, done, and optionally progress. The dashboard card renders whatever it's given. Store its state on the user record with a day key compared against todayKey().

Adding a currency source

Award server-side, cap it per day, and read the balance back from the response rather than trusting the client's arithmetic.

Deploying

Pushing to main triggers a Railway build; the site is live at angeldom.me within a minute or two. SESSION_SECRET must be set in production or sessions reset on every deploy. DATA_DIR must point at the mounted volume or data is lost on redeploy.

Because data files are gitignored, a deploy never touches live data — and local experiments never reach production. Verify a deploy landed by fetching something the new code serves, not just by checking the site responds.

Known limits

  • Sessions are in-memory — every restart signs everyone out.
  • JSON files don't scale and have no transactions. Fine for a handful of accounts; swap for a database before it's more.
  • Avatars are base64 inside users.json, which is why they're shrunk to 256px. Many large images would bloat every read of the file.
  • Snake and Writing are client-refereed — see the trust notes above.
  • No rate limiting on login, so passwords could be brute-forced given time.
  • Ten copies of the nav and theme block. The single biggest source of drift; a shared include would repay itself.
